# Notes

Личные заметки: доступ с нескольких устройств, PostgreSQL, использование как портфолио в резюме.

**Уровень продукта:** по сути это уже **Structured Knowledge Graph Lite** (Knowledge Engine), а не просто заметки: reference blocks, потенциальные backlinks, блочная модель. Это плюс для портфолио. **Phase 1 держи маленькой** — базовый функционал без раздувания.

**Цели:**
- Один аккаунт (только владелец). Гостевых аккаунтов нет — гости просто просматривают публичные заметки без входа.
- Видимость заметки — **enum** (не boolean): `PRIVATE` (по умолчанию), `PUBLIC`; позже возможно: unlisted, share link, pinned portfolio.
- Иерархия: папки → вложенные папки (любая глубина) → внутри папок заметки (как файлы).

**Стек:** Express + TypeScript, Angular, PostgreSQL.  
См. `PROJECT_STRUCTURE.md`, `AI_INSTRUCTIONS.md`.

## Модель данных

### Spaces (пространства)

- Таблица **spaces**: `id`, `name`, `root_folder_id`, **`about_note_id`** (nullable, FK на notes).
- При создании space создаётся только корневая папка и запись в `spaces` с `about_note_id = NULL`; отдельная заметка «About» при создании не создаётся. При необходимости about можно задать позже (пока только через БД или будущий API).

### Папки

- Дерево без ограничения глубины. **Путь:** в БД — **path** (ltree или materialized path). Два решения заранее:
  - **Сегмент пути:** использовать **folder id** (например `f.1234` в ltree), не slug — slug может меняться, id нет.
  - **Каскад при перемещении:** при move папки нужно обновить path у всех потомков; с ltree это один UPDATE по prefix — заложить в логику move.

### Заметки и блоки (строгая модель)

- **Заметка** — метаданные: id, folder_id, title, visibility (enum: `PRIVATE` | `PUBLIC`; позже: unlisted, share link, pinned portfolio), createdAt, updatedAt.
- **Контент заметки — не один JSON blob.** Строгая модель:
  - **Note** → массив блоков `blocks[]`, каждый блок с полем `type`.
  - Типы блоков: `text` | `chart` | `db_schema` (и при необходимости другие).
- **Хранение в БД:** таблица **note_blocks**:
  - `id`, `note_id`, `type`, **`position`**, **`version`** (int), `data` (jsonb).
  - **position:** **numeric(20,10)** (не float — накопление ошибок, странности сортировки). Вставка «между a и b» — новая позиция из того же типа. **Правило rebalance:** если «масштаб» позиций переходит порог (например дробная часть разрастается: 1.0000000001, 1.0000000002 через много вставок) — **IF scale(position) > threshold → rebalance**. Раз в N операций или по порогу переиндексировать в 1.0, 2.0, 3.0… Заранее решает дрейф к numeric chaos через год.

### Код в блоках (highlight.js)

- **Хранить только** `raw_code` (исходный код в `data` блока).
- **Подсветка** — только на уровне view: при отображении рендерить через highlight.js, пользователь видит подсвеченный вывод; в БД — только raw.

### Поиск по контенту (без FTS по целому JSON)

- FTS по всему JSON заметки даёт грязный и медленный поиск.
- **Решение:** колонка **searchable_tsv** типа **tsvector** на уровне заметки. Backend агрегирует в tsvector:
  - текст из text-блоков,
  - raw_code из code-блоков,
  - заголовки (title заметки и при необходимости блоков).
- **Пересчёт searchable_tsv — гибридный (без лишней инфры с нуля):** для **маленьких** заметок (например до N символов агрегированного текста) — пересчитывать **синхронно** (UX без лагов). Если объём больше N → **search_dirty = true** и пересчёт в **фоне**. Так не усложняем раньше времени воркерами и точками отказа.

### Редактор и ссылки

- **TipTap и блоки (практичный компромисс Phase 1):** блоки остаются source of truth, но не «редактор редактора». Для **text**-блока хранить **tiptap-json только этого блока**. Для **code**-блока — **raw_code**. Ссылки на заметки — структурированные marks/nodes внутри tiptap-json с **target_note_id**. Так не переизобретаем редактор, а TipTap остаётся view layer над блоками.
- **Ссылка на заметку = Reference Block (embed):** отдельный тип блока и block renderer. В **note_blocks**: `type = "note_link"`, `data = { target_note_id, display: "card" | "inline" }`. На UI — компонент **`<app-note-reference-card />`**: заголовок, иконка, путь папки, стиль, опционально hover preview. Это не только UI: заложена основа **graph relations** (связи между заметками). Опционально позже — таблица **note_relations**: `source_note_id`, `target_note_id` (даже если не используешь сразу: backlinks, граф ссылок строятся на этом). Reference block = потенциальный hidden gem для Knowledge Graph.

## Поиск

Два уровня:

1. **По папкам и заметкам** — по названиям папок и заголовкам заметок; клик — к папке или открытие заметки.
2. **По контенту заметок** — FTS по **searchable_tsv** (tsvector) даёт **какие заметки подходят**. Чтобы сделать **scroll-to-first-match**, точное место совпадения не хранить в БД: на **клиенте** после открытия заметки пробежаться по загруженным блокам и найти первое совпадение → scroll. То есть tsvector — для фильтра заметок; позиция первого совпадения — на фронте по блокам.

## Боковая панель (Sidebar)

Структура боковой панели — логичная и расширяемая.

### Основные элементы

**1. About App**

- Отдельная кнопка/пункт в сайдбаре.
- По структуре — обычная заметка (аналог стартовой страницы в Confluence).
- Используется как **стартовая страница приложения** при открытии или по умолчанию.

**2. Recent**

- Раздел с выпадающим списком (со стрелкой раскрытия/сворачивания).
- Показывает **только недавно открытые заметки** (не spaces).
- Порядок и состав — по дате/времени последнего открытия.
- Хранение: `RecentService`, ключ в localStorage.

**3. Favourite**

- Раздел с **избранным**: закреплённые **пространства (Spaces)**, папки и заметки.
- В списке порядок: сначала **spaces**, затем папки, затем заметки.
- Клик по space ведёт на `/home?space=id` — сайдбар раскрывает этот space.
- Хранение: `FavouriteService`, в т.ч. `spaceIds: number[]` в localStorage.

### Пространство (Space) — вместо «Folders»

- Концепция **Space** (пространство) — основной контейнер содержимого.
- Список spaces с раскрытием: при клике по названию space раскрывается его контент.
- **Строка space:** название + шеврон (раскрыт/свёрнут) + **кнопка троеточие (⋯)** справа. Вся строка — одна зона: общий hover и общий вид «открыт» (expanded), включая кнопку троеточия.
- Внутри раскрытого space отображаются два блока на одном уровне (единый стиль, иконки слева):

  **About (если есть)**  
  - Одна заметка пространства, **не под** блоком Content. Показывается только если у space задан `about_note_id`.
  - Иконка «About» слева (не меняется при hover). Подпись: «About space {имя space}». Клик открывает эту заметку в основной области.
  - При создании **нового** space заметка About **не создаётся** (в БД `spaces.about_note_id` допускает NULL).

  **Content**  
  - Иконка документа слева; при hover по строке (включая кнопку «+») иконка меняется на chevron-down. Справа — кнопка **+** (создать папку или заметку).
  - Ниже — дерево папок и список заметок выбранной папки.

- В **конце списка** spaces — кнопка **Create** (новый space). По нажатию — модалка ввода имени; создаётся корневая папка и запись space без about-заметки.

**Контекстное меню space (троеточие):**

- По клику на троеточие открывается меню (рендер в **fixed overlay**, поверх всего, позиция по кнопке).
- Пункты: **Edit** (редактирование имени space), **Favourites** (звезда: добавить/убрать space из избранного), разделитель, **Delete** (красный текст, иконка корзины).
- Delete: модалка с полем ввода; подтверждение только после ввода слова «delete» и кнопка Delete (красная).
- Edit: модалка с полем имени и кнопками Cancel / Update.

## Drag and drop (CDK Drag) — только папки и заметки

**Два уровня перемещения** (не три). **Не смешивать в один generic handler.** Два сервиса:

| Уровень | Сервис | Что меняется |
|--------|--------|----------------|
| 1️⃣ Folder → Folder | **FolderDragService** | path (ltree), каскад потомков |
| 2️⃣ Note → Folder | **NoteDragService** | folder_id + order заметки |

**Внутри заметки (редактор) cdkDrag не использовать** — это осознанное архитектурное решение, см. ниже.

### Не обновлять БД на каждый drop

- ❌ Плохо: drop → сразу PATCH API → reorder → refresh (много drag подряд, сетевые лаги, рассинхрон).
- ✔️ Паттерн: **drop → локально меняем position → UI обновляется мгновенно → debounce(300–500 ms) → batch update API.**

### Drag preview

- **FolderDragPreviewComponent**, **NoteDragPreviewComponent**. Концепция DragPreviewService; CDK DragPreview — быстрее ощущение drag, не перерисовывается весь DOM.

### Nested drop lists (папки → вложенные папки → заметки)

- **cdkDropListGroup** и **динамический connectedTo**. Drop target знает тип: **data-drag-type = "folder" | "note"** (только эти два — блоки в редактор не через CDK).

### Backend: move как domain action

- **PATCH /notes/move** с телом `{ note_id, new_folder_id, new_position }`. Не PATCH /notes/:id.

### UX

- **Drag note → hover по папке → папка auto-expand через ~400 ms** (Notion-feel). Hover-delay через RxJS.

## Reorder блоков внутри заметки — без CDK Drag

**cdkDrag внутри редактора заметки не использовать.** Не упрощение, а зрелый выбор.

### Почему TipTap и CDK Drag плохо совместимы

- CDK Drag: DOM reorder, placeholder, transform. TipTap: сам управляет DOM, internal state, selection.
- Результат: курсор прыгает, selection ломается, undo ведёт себя странно, rerender ломает drag preview. Классическая проблема ProseMirror-based редакторов.
- При drag через CDK: DOM меняется → TipTap считает, что пользователь изменил структуру → editor пересобирает state → позиция drop теряется, ghost updates.

### Как делают зрелые редакторы (Notion, Obsidian, Craft)

- Не используют drag-framework внутри редактора. **Handle слева** → reorder через **editor commands** (handle drag → editor transaction → reorder blocks), а не DOM-drag.

### Наш путь: блоки = source of truth, position numeric

- Reorder блоков внутри заметки: **кнопки ↑ ↓**, **slash menu**, **"move block"**. Команды редактора, обновление `note_blocks.position` ((prev+next)/2). **Rebalance по порогу:** если scale(position) > threshold → rebalancePositions(note_id).
- Стабильно, предсказуемо, без конфликтов с TipTap.

## План

### Фаза 1 — Основа и хранилище

- [ ] **Backend:** PostgreSQL (клиент, env, миграции/init-скрипт).
- [ ] **Backend:** папки — id, parent_id, title, порядок, **path** (ltree). Сегмент пути = **folder id** (не slug). При **move** папки — один UPDATE по prefix для обновления path у всех потомков.
- [ ] **Backend:** заметки — id, folder_id, title, visibility enum, createdAt, updatedAt, **searchable_tsv** (tsvector), **search_dirty**. Пересчёт tsv: **гибрид** — синхронно для маленьких заметок (до N символов), иначе search_dirty + фоновый пересчёт.
- [ ] **Backend:** таблица **note_blocks** — id, note_id, type, **position** numeric(20,10), version, data (jsonb). **Rebalance:** IF scale(position) > threshold → rebalancePositions(note_id) (переиндекс 1.0, 2.0, …). Опционально (для графа/backlinks): таблица **note_relations** — source_note_id, target_note_id; заложить даже если не использовать сразу.
- [ ] **Backend:** CRUD папок (дерево, path, каскад при move). CRUD заметок и блоков. **PATCH /notes/move** с телом `{ note_id, new_folder_id, new_position }` (move как domain action, не PATCH /notes/:id). Опционально: rebalance позиций блоков (rebalancePositions) по запросу или раз в N операций.
- [ ] **Frontend:** сервис(ы) для API папок, заметок, блоков.
- [ ] **Frontend:** дерево папок + список заметок; создание/редактирование заметки, переключение visibility.
- [ ] **Frontend:** редактор **TipTap** — text-блок: tiptap-json этого блока; code-блок: raw_code; ссылки в tiptap-json как marks/nodes с target_note_id. Блок **note_link** — рендер компонентом **&lt;app-note-reference-card&gt;** (или inline): заголовок, иконка, путь папки, стиль, опционально hover preview; не &lt;a&gt;.
- [ ] **Frontend:** scroll-to-first-match: после открытия заметки по результату поиска — локально пройти по блокам, найти первое совпадение, проскроллить.
- [ ] **Frontend (CDK Drag):** только **FolderDragService** и **NoteDragService** (папки и заметки; внутри заметки cdkDrag не использовать). Drop → локальное обновление → UI → debounce 300–500 ms → batch API. Превью: FolderDragPreviewComponent, NoteDragPreviewComponent. Nested lists: cdkDropListGroup, connectedTo, **data-drag-type = "folder" | "note"**. UX: hover по папке → auto-expand через ~400 ms (RxJS).
- [ ] **Frontend (reorder блоков в заметке):** без cdkDrag. Handle слева + editor commands (↑↓, slash menu, move block). position: (prev+next)/2; при scale > threshold — rebalance. Стабильно с TipTap.
- [ ] **Frontend (боковая панель):** **About App** — кнопка, открывает стартовую заметку (обычная заметка, по умолчанию или из настроек). **Recent** — выпадающий список (со стрелкой) недавно открытых заметок. **Favourite** — выпадающий список избранного (закреплённые папки/заметки). **Space** — вместо заголовка «Folders»: контейнер со списком (папки + заметки выбранной папки), в конце кнопка **Create** с выбором: заметка, папка (позже — другие типы).
