# Архитектура данных: «правильная» файловая система в приложении

Референс: структура хранения, логика доступа, масштабирование, поиск, версии, безопасность, работа с ИИ.  
Если ИИ «тупит», чаще всего проблема в структуре данных.

---

## 1. Что такое «правильная» система

Не только папки и файлы, а:

- **Структура хранения** — явная, предсказуемая
- **Логика доступа** — по ролям/space, не «по папке»
- **Масштабирование** — adjacency list, индексы
- **Поиск** — отдельный слой (FTS, AI-index)
- **Версии** — хотя бы на уровне контента/блоков
- **Безопасность** — space-level + override на документ
- **Работа с ИИ** — разделение metadata / content / AI-index, chunking

---

## 2. Рекомендуемая база: иерархия + метаданные

- **Не** физическая вложенность на уровне БД.
- **Adjacency list:** у сущности есть `parent_id` (nullable), при необходимости `space_id`, `owner_id`, типы.
- Одна логическая таблица на тип сущности (или единая `entities` с `type`), простые связи.

---

## 3. Уровни в нашем приложении

| Уровень | Сущность | Назначение |
|--------|----------|------------|
| 1 | **Space** | Изолированный контейнер (owner, permissions, security) |
| 2 | **Structure** | Внутри space: folder (parent_id) и note (folder_id) |
| 3 | **Content** | Блоки заметки, версии, отдельно от метаданных |

Логически:

```
space
 ├── folder (root = root_folder_id)
 │    ├── folder
 │    │    └── note
 │    └── note
 └── note (в корне = в root folder)
```

В БД это записи с `parent_id` (папки) и `folder_id` (заметки).

---

## 4. Текущая схема vs рекомендации

| Рекомендация | Текущее состояние |
|--------------|------------------|
| **Adjacency list** | ✅ `folders.parent_id`, `notes.folder_id` (родитель заметки = папка) |
| **Path для запросов** | ✅ `folders.path` (ltree) — денормализация для быстрых выборок и move |
| **Space как контейнер** | ✅ `spaces.root_folder_id` — принадлежность папки к space через корень |
| **Тип сущности** | ✅ Отдельные таблицы `folders`, `notes` (явный тип без полиморфных type) |
| **Метаданные vs контент** | ✅ `notes` — метаданные; `note_blocks` — контент (type, position, data jsonb) |
| **Версии** | ✅ `note_blocks.version` |
| **Поиск** | ✅ `notes.searchable_tsv` (tsvector), `search_dirty` для отложенного пересчёта |
| **Доступ** | ✅ `visibility` на note/space; единый владелец (Phase 1) |
| **AI-friendly** | ⚠️ Контент по блокам есть; chunking/embedding per chunk — при необходимости позже |
| **Права** | ⚠️ effective_permission = document ?? parent ?? space — при появлении ролей |

---

## 5. Ошибки, из-за которых ИИ плохо работает

- Нет явного типа сущности → у нас тип задан таблицей (folder/note) и полем в блоке (type).
- Нет разделения контента и метаданных → есть: notes (meta) + note_blocks (content).
- Документ слишком большой → контент разбит на блоки.
- Нет версии → version в блоках.
- Нет нормальной индексации → searchable_tsv, индексы по folder_id, path.

---

## 6. Куда двигаться при усложнении

- **Права:** space-level + override на документ, inheritance (effective_permission).
- **ИИ:** chunking контента, отдельная таблица/индекс под embeddings (per chunk), без «сырой портянки».
- **Файлы (если появятся):** storage abstraction (S3-style), в БД только reference (storage_path, mime_type, size, owner, space_id).

---

## 7. Кратко

- Одна логическая таблица на тип сущности, `parent_id`/`folder_id` для иерархии.
- Строгий слой прав на уровне space с переопределением на документ.
- Версионирование контента (блоки).
- Разделение: метаданные / контент / поиск (и при необходимости AI-index).
- Текущая схема уже следует этим принципам в рамках Phase 1; расширение — без ломания базы.
